# 今日學習目標
學會寫給 AI 看的「說明書」(MRD & PRD)。
理解什麼是「上下文工程」(Context Engineering)。
實踐「人機協作循環」(Human-in-the-loop)。

# 回顧第一天：我們做了什麼？
我們學會了跟 AI 聊天。
我們做出了一個簡單的 WBC 棒球 App。
問題： 如果我們要做出更複雜、更有用的工具，光靠聊天夠嗎？

# 蓋房子不需要先畫圖嗎？
亂試指令 = 拿著磚頭直接疊。
寫計畫書 = 先畫好建築設計圖。
AI 時代的關鍵： 你的設計圖越細，AI 蓋出來的房子就越穩。

# 什麼是 MRD？(市場需求文件)
大白話： 「為什麼要蓋這座房子？誰要來住？」
在教學上：你的學生是誰？他們遇到什麼學習困難？

# MRD 的核心內容：解決誰的痛苦？
目標對象 (Audience)：學生、家長還是老師自己？
核心問題 (Pain Point)：例如：學生看不懂複雜的棒球數據。
解決方案 (Solution)：一個直覺的數據視覺化工具。

# 練習 1：寫下你的 MRD 核心（3 分鐘）
請在紙上寫下一句話：
「我要為 [誰] 解決 [什麼問題]，因為這能帶來 [什麼好處]。」

# 什麼是 PRD？(產品需求文件)
大白話： 「房子要有幾間房？要有電梯嗎？插座在哪？」
在教學 App 上：有哪些按鈕？點下去會發生什麼事？

# PRD 的結構：給 AI 的施工圖
功能列表 (Features)：有哪些按鈕、輸入框。
使用者路徑 (User Flow)：第一步做什麼，第二步看到什麼。
成功標準：App 跑出什麼結果才算成功？

# 為什麼要用 .md (Markdown) 格式寫？
結構清晰：AI 最喜歡看 # 標題、* 列表。
輕量好改：不用管排版，專注在內容。
AI 友好：這是 2026 年程式開發的共通語言。

# 技術平權：讓 AI 幫你寫 PRD
不需要自己從頭寫。
你只要把「想法」餵給 AI，跟它說：「請幫我把這個想法整理成一份專業的 PRD.md」。

# 實作練習：生成你的第一份 PRD
任務：對AI輸入你的 MRD 想法，要求它產出 PRD。
觀察：看看 AI 是不是比你想像中更懂你的需求？

# 老師的角色轉變：從「寫作者」到「總編輯」
AI 寫出的 PRD 可能有錯。
老師負責「審核」與「修改」。這就是接下來要談的核心概念。

# 什麼是上下文工程 (Context Engineering)？
想像你在教一個新來的助教。
如果只說「去改作業」，他會一頭霧水。
如果說「這是去年的標準，這是今年的重點，請參考學生的背景...」，這就是上下文。

# 上下文工程的三大支柱
角色背景 (Role)：你是誰？（專業工程師、熱血教練）。
具體約束 (Constraint)：不能用什麼？（不要用深奧的術語、不要超過五頁）。
參考範例 (Few-Shot)：以前成功的案例長什麼樣？

# 為什麼上下文決定了 AI 的智商？
垃圾進，垃圾出 (Garbage In, Garbage Out)。
高品質的背景資訊 = 高質量的程式碼產出。

# 進入核心：Human-in-the-loop (HITL)
大白話： 「AI 做，人驗收，不對再改」的華爾滋。
這是 2026 年開發互動網頁程式的標準動作。

# HITL 的四個步驟
指令 (Prompt)：老師發起。
生成 (Generation)：AI 產出代碼或文件。
驗收 (Review)：老師檢查，發現按鈕太小或功能不對。
修正 (Refine)：老師對 AI 說「請把按鈕改大一點」。

# 為什麼老師適合做 HITL 的領航員？
老師最擅長的是「批改」與「給建議」。
把 AI 當成學生，你在批改它的「程式碼作業」。

# 技術難題怎麼辦？「請 AI 教我」
如果 AI 報錯了，老師看不懂怎麼辦？
技巧：把錯誤訊息貼回給 AI，說：「我不懂這行程式碼，請用白話文解釋並幫我修正。」

# 實作挑戰：WBC App 進化論
拿出第一天的 App 雛形。
使用 PRD 概念，要求 AI 增加一個功能（例如：加入 Lucide 的星星圖示）。
經歷至少兩次「AI 做 -> 我改 -> AI 再做」的循環。

# 深度思考：這不只是開發程式
這種「提問-反饋-修正」的過程，其實就是最優秀的教學過程。
你在學習如何更精確地表達想法。
